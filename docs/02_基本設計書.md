## 4. API設計

### 4.0 事業者・アカウント登録 API（認証不要）

| メソッド | パス | 説明 | ボディ |
|---------|------|------|--------|
| POST | /api/businesses | 事業者登録。事業者IDを発行しメール送信 | { companyName, address, contactName, phone, email } |
| POST | /api/register | アカウント新規登録。事業者の最初の登録者はADMIN、以降STAFF | { businessCode, email, name, password, passwordConfirm } |
| POST | /api/password-reset | パスワード再設定申請。該当ユーザに再設定URLをメール送信 | { email } |
| POST | /api/password-reset/confirm | トークン検証のうえパスワードを更新 | { token, password } |

#### 事業者情報 API（認証必要・自社のみ）

| メソッド | パス | 説明 | ボディ |
|---------|------|------|--------|
| GET | /api/businesses/:id | 事業者詳細（自社のみ。他社は404） | - |
| PATCH | /api/businesses/:id | 事業者情報更新（管理者・自社のみ） | { companyName, address, contactName, phone, email } |

- 電話番号は登録時と同じ形式チェック（半角数字とハイフン、0始まり10〜11桁）。

- `/api/businesses`: 一意な `code` を生成して `businesses` を作成、`sendBusinessIdEmail()` で事業者IDを送付し、`code` をレスポンスに含める。
- `/api/register`: `businessCode` から事業者を特定（不在は404）、メール重複は409、パスワード不一致は400。所属事業者のユーザ数が0ならADMIN、それ以外はSTAFF。

#### パスワード再設定（忘れた場合）

- `/api/password-reset`: メールアドレスからユーザを特定し、ランダムトークン（SHA-256ハッシュを `password_reset_tokens` に保存、有効期限1時間）を発行して再設定URLをメール送信（`sendPasswordResetEmail()`）。**メールアドレス探索を防ぐため、ユーザの存在有無に関わらず常に成功を返す。**
- `/api/password-reset/confirm`: トークンのハッシュで検証（不在・期限切れ・使用済みは400）。検証成功でパスワードを更新し、トークンを使用済みにする（トランザクション）。
- ログイン中の本人によるパスワード変更（現在のパスワード検証あり）とは別系統。こちらはログイン不可のユーザ向け。

### 4.4 ユーザ管理 API（管理者のみ／自身は一部操作可）

| メソッド | パス | 説明 | ボディ |
|---------|------|------|--------|
| GET | /api/users | 同一事業者のユーザ一覧（管理者） | - |
| POST | /api/users | ユーザ新規作成（管理者） | { name, email, password, role } |
| GET | /api/users/:id | ユーザ詳細（本人または管理者） | - |
| PATCH | /api/users/:id | ユーザ更新（本人または管理者） | { name?, email?, currentPassword?, password?, role? } |
| DELETE | /api/users/:id | ユーザ削除（管理者。自分自身は不可） | - |

- ロール変更は管理者のみ。
- パスワード変更は**本人のみ**可能。本人の変更時は `currentPassword`（現在のパスワード）の検証が必要（不一致は422）。管理者であっても他ユーザのパスワードは変更不可（`password` 指定時は403）。
- メール重複は409。対象ユーザが別事業者の場合は404。

### 4.5 テナント分離方針

- 認証済みAPI／サーバコンポーネントの `users`・`care_recipients` への読み書きは、すべてセッションの `businessId` で絞り込む（一覧は `where: { businessId }`、作成は `data` に付与、個別取得・更新・削除は所属事業者一致を確認し、不一致は404）。
- `session.user.businessId` は NextAuth の jwt / session コールバックで伝播する。

## 5. ウェアラブルデバイス連携設計（将来実装）

### 5.1 技術方式
- **プロトコル**: Bluetooth Low Energy (BLE)
- **API**: Web Bluetooth API（ブラウザネイティブ）
- **必要条件**: HTTPS環境、Chrome系ブラウザ

### 5.2 対応デバイスプロファイル
- Health Thermometer Profile (体温計)
- Heart Rate Profile (心拍計)
- Blood Pressure Profile (血圧計)
- Pulse Oximeter Profile (SpO2計)

### 5.3 実装方針
1. バイタル入力フォームに「デバイスから取得」ボタンを配置（現状はdisabled）
2. クリック時にWeb Bluetooth APIでデバイスをスキャン
3. 取得値をフォームに自動入力（ユーザが確認後に保存）
4. `input_method = WEARABLE` で記録
