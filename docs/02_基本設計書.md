## 4. API設計

### 4.1 事業者・アカウント登録 API（認証不要）

| メソッド | パス | 説明 | ボディ |
|---------|------|------|--------|
| POST | /api/businesses | 事業者登録。事業者IDを発行しメール送信 | { companyName, address, contactName, phone, email } |
| POST | /api/register | アカウント新規登録。事業者の最初の登録者はADMIN、以降STAFF | { businessCode, email, name, password, passwordConfirm } |
| POST | /api/password-reset | パスワード再設定申請。該当ユーザに再設定URLをメール送信 | { email } |
| POST | /api/password-reset/confirm | トークン検証のうえパスワードを更新 | { token, password } |

#### 事業者情報 API（認証必要・自社のみ）

| メソッド | パス | 説明 | ボディ |
|---------|------|------|--------|
| GET | /api/businesses/:id | 事業者詳細（自社のみ。他社は404） | - |
| PATCH | /api/businesses/:id | 事業者情報更新（管理者・自社のみ） | { companyName, address, contactName, phone, email } |

- 電話番号は登録時と同じ形式チェック（半角数字とハイフン、0始まり10〜11桁）。

- `/api/businesses`: 一意な `code` を生成して `businesses` を作成、`sendBusinessIdEmail()` で事業者IDを送付し、`code` をレスポンスに含める。
- `/api/register`: `businessCode` から事業者を特定（不在は404）、メール重複は409、パスワード不一致は400。所属事業者のユーザ数が0ならADMIN、それ以外はSTAFF。

#### パスワード再設定（忘れた場合）

- `/api/password-reset`: メールアドレスからユーザを特定し、ランダムトークン（SHA-256ハッシュを `password_reset_tokens` に保存、有効期限1時間）を発行して再設定URLをメール送信（`sendPasswordResetEmail()`）。**メールアドレス探索を防ぐため、ユーザの存在有無に関わらず常に成功を返す。**
- `/api/password-reset/confirm`: トークンのハッシュで検証（不在・期限切れ・使用済みは400）。検証成功でパスワードを更新し、トークンを使用済みにする（トランザクション）。
- ログイン中の本人によるパスワード変更（現在のパスワード検証あり）とは別系統。こちらはログイン不可のユーザ向け。

|---------|------|------|--------|
| GET | /api/care-recipients | 一覧取得（削除済除く） | - |
| POST | /api/care-recipients | 新規作成 | CareRecipientInput |
| GET | /api/care-recipients/:id | 詳細取得 | - |
| PATCH | /api/care-recipients/:id | 更新 | Partial<CareRecipientInput> |
| DELETE | /api/care-recipients/:id | 論理削除 | - |

### 4.3 バイタル API

| メソッド | パス | 説明 |
|---------|------|------|
| GET | /api/care-recipients/:id/vitals | 一覧取得（最新500件、新しい順） |
| POST | /api/care-recipients/:id/vitals | 記録作成 |
| PATCH | /api/care-recipients/:id/vitals/:vitalId | 記録更新（**最新の1件のみ**） |

- GET はクエリ `days` で取得期間を絞り込む。`days=30`（既定）/ `7` / `180` / `365` のように日数を指定、`days=all`（または不正値）で全期間。グラフの期間セレクタが利用する。ダッシュボードは `days=all` で取得し、クライアント側で期間フィルタする。
- PATCH は対象バイタルが同一事業者の対象者のものか確認し（不一致は404）、その対象者の**最新バイタルでない場合は編集不可**（403, `code: NOT_LATEST`）。死亡日時・退院日時は送信値で上書き（未指定は null クリア）。
- バイタルの削除（DELETE）は提供しない。

### 4.4 家族 API

| メソッド | パス | 説明 |
|---------|------|------|
| GET | /api/families | 全家族一覧取得 |
| POST | /api/families | 家族新規作成 |
| PATCH | /api/families/:id | 家族情報更新 |
| DELETE | /api/families/:id | 家族を物理削除（`?careRecipientId=` 現在の対象者。他対象者に紐づく場合は409） |
| POST | /api/care-recipients/:id/families | 介護対象者へ家族を紐付け |
| DELETE | /api/care-recipients/:id/families/:familyId | 紐付け解除 |

### 4.5 ユーザ管理 API（管理者のみ／自身は一部操作可）

| メソッド | パス | 説明 | ボディ |
|---------|------|------|--------|
| GET | /api/users | 同一事業者のユーザ一覧（管理者） | - |
| POST | /api/users | ユーザ新規作成（管理者） | { name, email, password, role } |
| GET | /api/users/:id | ユーザ詳細（本人または管理者） | - |
| PATCH | /api/users/:id | ユーザ更新（本人または管理者） | { name?, email?, currentPassword?, password?, role? } |
| DELETE | /api/users/:id | ユーザ削除（管理者。自分自身は不可） | - |

- ロール変更は管理者のみ。
- パスワード変更は**本人のみ**可能。本人の変更時は `currentPassword`（現在のパスワード）の検証が必要（不一致は422）。管理者であっても他ユーザのパスワードは変更不可（`password` 指定時は403）。
- メール重複は409。対象ユーザが別事業者の場合は404。

### 4.6 お問い合わせ API（認証必要）

| メソッド | パス | 説明 | ボディ |
|---------|------|------|--------|
| POST | /api/contact | ログインユーザからシステム管理者へ問い合わせメールを送信 | { subject, message } |

- 送信者名・メールアドレスはセッションから取得する（本文に含める）。`sendContactEmail()` で送信し、SMTP未設定時はサーバログ出力にフォールバックする。

### 4.7 テナント分離方針

- 認証済みAPI／サーバコンポーネントの `users`・`care_recipients`・`families` への読み書きは、すべてセッションの `businessId` で絞り込む（一覧は `where: { businessId }`、作成は `data` に付与、個別取得・更新・削除は所属事業者一致を確認し、不一致は404）。
- 家族の紐付け／解除（`/api/care-recipients/:id/families`）は、対象者・家族の双方が自事業者に属することを検証してから実行する（越境IDORの防止）。
- `session.user.businessId` は NextAuth の jwt / session コールバックで伝播する。

## 5. SEO・メタデータ設計

医療・個人情報を扱う認証配下は検索エンジンに出さず、**公開ページ（ログイン・事業者登録・アカウント登録・パスワード再設定申請）だけを集客対象**とする方針。

- **既定で noindex**: ルート `app/layout.tsx` の `metadata` で `robots: { index: false }` を既定とし、認証配下・API・未指定ページをまとめてインデックス拒否。
- **公開ページのみ index 許可**: 各公開ルートの `layout.tsx` で `robots: { index: true, follow: true }` ＋ タイトル・説明・`canonical`・OGP を個別付与。`reset-password/confirm` はトークン経由のため `noindex`。
- **robots.txt（`app/robots.ts`）**: 公開ページを許可、`/api/`・`/dashboard`・`/care-recipients`・`/users`・`/businesses`・`/contact`・`/reset-password/confirm` を Disallow、`sitemap.xml` を参照。
- **sitemap.xml（`app/sitemap.ts`）**: 公開4ページを優先度付きで列挙。
- **OGP画像（`app/opengraph-image.tsx`）**: `next/og` で動的生成（SNS共有用）。
- **共通定数（`lib/site.ts`）**: `SITE_URL`（`NEXT_PUBLIC_SITE_URL` → `NEXTAUTH_URL` → localhost の順でフォールバック）・`SITE_NAME`・`SITE_DESCRIPTION` を集約し、メタデータ/robots/sitemap で共有。
- **middleware**: `robots.txt`・`sitemap.xml`・`opengraph-image` を公開パス（認証除外）に追加し、未認証クローラが取得できるようにする。

> `NEXT_PUBLIC_SITE_URL` はビルド時に埋め込まれるため、本番（Vercel等）では環境変数に本番ドメインを設定する。

## 6. デプロイ・環境変数

本番（Vercel 等）では以下を環境変数に設定する。`.env` は gitignore 済みで、`.env.example` がテンプレート。

| 変数 | 必須 | 説明 |
|------|------|------|
| `DATABASE_URL` | ○ | PostgreSQL接続。ホスト型は `?sslmode=require` を付与 |
| `NEXTAUTH_SECRET` | ○ | セッションJWTの署名鍵。`openssl rand -base64 32` 等で強いランダム値を生成。未設定/プレースホルダは本番起動時にエラー（`lib/env.ts`） |
| `NEXTAUTH_URL` | ○ | 公開URL（末尾スラッシュ無し）。認証コールバック・パスワード再設定リンクに使用 |
| `NEXT_PUBLIC_SITE_URL` | △ | 集客SEO/OGP/sitemap用の絶対URL。**ビルド時に埋め込まれる**ため設定後は再デプロイが必要 |
| `SMTP_HOST`/`SMTP_PORT`/`SMTP_USER`/`SMTP_PASS`/`SMTP_FROM` | △ | メール送信（Brevo SMTP）。未設定時はサーバログ出力にフォールバック |
| `SUPPORT_EMAIL` | △ | お問い合わせの送信先 |

- 認証関連の必須変数は `lib/env.ts` の `assertProductionAuthEnv()` で本番起動時に検証し、未設定/プレースホルダなら**フェイルクローズ**（起動失敗）する。
- パスワード再設定等のメールリンクの絶対URLは `NEXTAUTH_URL` → `NEXT_PUBLIC_SITE_URL` → `VERCEL_URL` → localhost の順でフォールバック（`lib/site.ts` の `APP_URL`）。
